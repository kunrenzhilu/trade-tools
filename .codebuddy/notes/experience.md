# 编程原则

> 每次写代码前默念一遍。

---

## 1. 静默失败是调试成本最高的 bug

- `except Exception: return []`、`save=True` 但路径是 None → 看起来成功了，N 步之后发现什么都没发生
- 原则：**任何可能失败的操作，要么成功，要么让调用方立刻感知到（raise / warn / 返回 Result 类型）**

## 2. I/O 边界必须校验内容，不能只校验"是否存在"

- 文件存在 ≠ 数据有效（空文件、0 字节、只有表头）
- 网络请求没抛异常 ≠ 拿到了正确数据（403 但返回空列表）
- 原则：**读完之后立刻校验：records 非空？行数合理？关键字段有值？**

## 3. 调用方要对被调用方的副作用前提负责

- `xxx.save()` 写入哪个路径？传入的参数是否正确？
- 临时对象的方法调用产生的副作用，后续消费方能拿到吗？
- 原则：**往下传参数之前，确认接收方的"成功条件"是什么，而不是假设它"自己会处理"**

## 4. 不要在错误的抽象层找根因

- 403 ≠ GFW，可能只是少了 User-Agent
- 原则：**现象和根因之间至少隔着一层。先缩小范围做对照实验（什么方式能复现？什么方式不能？），再下结论**

## 5. 单元测试无法替代集成测试

**背景**：Phase 5 有 457 个单元测试全部通过，但系统跑起来之后出现了 6 个 bug：

| bug | 为什么单元测试没发现 |
|-----|-------------------|
| `group_id=UNKNOWN`（未调分层计算） | Container 的单元测试 mock 了 UniverseManager，根本不走真实分组逻辑 |
| 读到空数据库 | 测试用 `:memory:` 或 tmp_path，从不接触真实 `~/.mytrader/market_data.db` |
| datetime tz 不兼容 | 测试构造的 Signal.timestamp 是 tz-naive，真实系统产生的是 tz-aware |
| RiskManager 属性名写错 | CandidateSelector 测试直接传 `AccountState`，从未实例化 RiskManager |

**根本原因**：测试覆盖了每个模块的内部逻辑，但**没有测试模块之间的拼接**。集成测试缺失意味着：每块砖都是好的，但建出来的墙可能是歪的。

**原则**：
- 每个"链路节点"写完后，必须写一个从真实入口到该节点的端到端集成测试
- 集成测试要用**真实数据路径**（非 mock），至少要用能代表真实状态的 fixture
- 具体到本项目：每个新 Sprint 完成后，跑 `--scan-now morning`（哪怕本地没数据），确认从入口到链路末端没有崩溃

## 6. 单元测试测"信号正确"，但不测"策略能否完成交易闭环"

**背景**：Iter #8 的 `rsi_trend_filter` 单元测试全绿（信号 ∈ {-1,0,1}、上升趋势不 SELL、下降趋势不 BUY），却在真实回测中对 5 只股票产生 **0 个出场信号**——入场条件（上升趋势）和出场条件（下降趋势）在趋势方向上互斥，仓位永远无法平仓，退化成滞后 buy-and-hold。最讽刺的是：`test_uptrend_only_buy` / `test_downtrend_only_sell` 这两个"通过"的测试，验证的恰恰是导致 Bug 的那个特性，把致命缺陷当成正确行为认证了。

**原则**：
- 策略测试必须包含**交易闭环测试**：给一段真实/半真实数据，跑一次回测，断言 `已平仓交易数 ≥ 阈值`、`win_rate 非 NaN`、`持仓暴露时间比例合理`
- 入场和出场条件不能在同一维度上互斥（否则买了卖不掉）。设计策略时先问：**"我这笔仓位靠什么信号平掉？这个信号在目标行情里会不会几乎不触发？"**
- `win_rate=NaN / 交易笔数≈0` 但 Sortino/收益却正常，是"仓位不平仓、靠盯市和末尾强平凑出收益"的强信号，必须当异常查

## 7. "被选中" ≠ "有效"；指标是过滤器，不是优化目标

**背景**：Iter #9 为了让 `rsi_trend_filter` 能入选，把选择器从 Sortino 排序改成样本内 Alpha 排序。结果坏策略确实进了 4/6 组权重（"被选中"），但样本外组合 alpha 从 -11% 崩到 -25%。换指标只是把样本内的 regime 运气暴露出来，放大了过拟合。

**原则**：
- 一个策略"能进权重"不代表"有效"。判断有效的标准是**样本外表现**，不是它在选择器里排第几
- **在同一段数据上同时做参数搜索 + 策略选择，无论用什么单指标（Sortino/Sharpe/Alpha）都必然过拟合**。选择必须用样本外指标（Walk-Forward 验证期），或专用 holdout（前 N 年调参、后 1 年选策略）
- 遇到"结果变差"时，先质疑**输入的策略/数据是否本身就坏**，再去调选择指标。别把"策略坏了"误诊成"指标选错了"（DeepSeek 就栽在这句上）

## 8. 排序前必须先过硬门槛：健全性 + 风险 + 正超额

**背景**：当前 `_run_group` 的选择逻辑允许：0 平仓交易的退化策略、alpha 全负时选"没那么差"的负 alpha、负 alpha 被 `max(alpha,0.01)` 归一化成等权。WF 4/4 pass 但组合 alpha=-25%，因为 WF gate 只校验 DD/Sortino 不校验 alpha。

**原则**：
- 排序（选 top-K）之前，先过**硬门槛过滤**，顺序：① 健全性（交易笔数/win_rate 非退化）→ ② 风险（DD≤20%）→ ③ 正超额（alpha>0 或 >benchmark_margin）→ 最后才在合规候选里排序
- **没有候选满足门槛时，正确动作是"空仓/降现金/回退 benchmark"，不是"矬子里拔将军"强行 top-K**
- 验收 gate（无论 WF 还是 paper 准入）必须校验**跑赢 benchmark（正 alpha）**，DD/Sortino 达标 ≠ 有正 alpha
- 归一化权重时，负分不能用 `max(x, ε)` 掩盖——那会把"都不好"变成"等权都要"

## 9. spec 的根因假设需要用数据验证，不能照单全收

**背景**：Iter #16 spec 假设"SPX 成分股 vs SPY 存在结构性近零 alpha（-1% ~ 0%）"，理由是"SPY 即 S&P 500 本身，交易 SPX 组件 vs SPY 相当于和自己赛跑"。基于此假设，将 alpha gate 从 alpha>0 放宽至 alpha>-2%。但实际 reoptimize 结果显示 SPX_mid_vol 的 9 策略 alpha 范围为 -4.69% ~ -13.84%，SPX_high_vol 为 -3.61% ~ -13.40%。-2% 阈值对 SPX 组完全不够。

**原则**：
- spec 的根因分析是基于直觉/经验，不一定准确。实施前应先用数据验证："当前 SPX 组的 alpha 分布到底是什么范围？"
- 阈值选择应基于实际 alpha 分布，而非理论假设。如果 SPX 组的 alpha 中位数是 -6%，-2% 阈值就是无效的
- 放宽阈值是"治标"，改进策略 alpha 才是"治本"。如果所有策略在 SPX 上都跑输 SPY 5-14%，问题不在 gate 阈值，而在策略逻辑不适配 SPX 成分股的低波动率特征
- in-sample 与 OOS 阈值应分层：in-sample 宽松（让候选进入 OOS），OOS 严格（avg>0 汇总门槛）。Iter #16 的 -2% in-sample vs Iter #13 的 -5% OOS floor 设计是正确的分层，但 -2% 对 SPX 组来说仍然太严
