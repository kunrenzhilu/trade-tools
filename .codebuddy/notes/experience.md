# 编程原则

> 每次写代码前默念一遍。

---

## 1. 静默失败是调试成本最高的 bug

- `except Exception: return []`、`save=True` 但路径是 None → 看起来成功了，N 步之后发现什么都没发生
- 原则：**任何可能失败的操作，要么成功，要么让调用方立刻感知到（raise / warn / 返回 Result 类型）**

## 2. I/O 边界必须校验内容，不能只校验"是否存在"

- 文件存在 ≠ 数据有效（空文件、0 字节、只有表头）
- 网络请求没抛异常 ≠ 拿到了正确数据（403 但返回空列表）
- 原则：**读完之后立刻校验：records 非空？行数合理？关键字段有值？**

## 3. 调用方要对被调用方的副作用前提负责

- `xxx.save()` 写入哪个路径？传入的参数是否正确？
- 临时对象的方法调用产生的副作用，后续消费方能拿到吗？
- 原则：**往下传参数之前，确认接收方的"成功条件"是什么，而不是假设它"自己会处理"**

## 4. 不要在错误的抽象层找根因

- 403 ≠ GFW，可能只是少了 User-Agent
- 原则：**现象和根因之间至少隔着一层。先缩小范围做对照实验（什么方式能复现？什么方式不能？），再下结论**

## 5. 单元测试无法替代集成测试

**背景**：Phase 5 有 457 个单元测试全部通过，但系统跑起来之后出现了 6 个 bug：

| bug | 为什么单元测试没发现 |
|-----|-------------------|
| `group_id=UNKNOWN`（未调分层计算） | Container 的单元测试 mock 了 UniverseManager，根本不走真实分组逻辑 |
| 读到空数据库 | 测试用 `:memory:` 或 tmp_path，从不接触真实 `~/.mytrader/market_data.db` |
| datetime tz 不兼容 | 测试构造的 Signal.timestamp 是 tz-naive，真实系统产生的是 tz-aware |
| RiskManager 属性名写错 | CandidateSelector 测试直接传 `AccountState`，从未实例化 RiskManager |

**根本原因**：测试覆盖了每个模块的内部逻辑，但**没有测试模块之间的拼接**。集成测试缺失意味着：每块砖都是好的，但建出来的墙可能是歪的。

**原则**：
- 每个"链路节点"写完后，必须写一个从真实入口到该节点的端到端集成测试
- 集成测试要用**真实数据路径**（非 mock），至少要用能代表真实状态的 fixture
- 具体到本项目：每个新 Sprint 完成后，跑 `--scan-now morning`（哪怕本地没数据），确认从入口到链路末端没有崩溃
