## 第七章：绩效评估与策略自我优化

> **核心理念**：所有存储在 `trade_log.json` 和 `positions.json` 中的历史交易数据，不仅用于绩效统计，更是策略自我优化的**核心燃料**。AI 必须主动挖掘交易记录中的规律和错误模式，形成可量化的优化建议，持续降低"系统性重复犯错"的概率。

---

### 7.1 实时绩效追踪

#### 7.1.1 核心绩效指标

| 指标 | 计算方式 | 预警线 | 触发动作 |
|------|----------|--------|----------|
| **当日盈亏** | Σ(已实现盈亏 + 浮动盈亏) | 亏损 > 2% 预警 | 在盘中监控报告中标黄 |
| **周累计收益率** | 本周五净值 / 上周五净值 - 1 | — | 纳入周报 |
| **最大回撤（滚动20日）** | (峰值净值 - 当前净值) / 峰值净值 | > **10%** 触发策略复查 | 暂停新开仓，生成复查报告 |
| **夏普比率（滚动60日）** | (策略收益 - 无风险收益) / 策略波动率 | < **1.0** 触发策略复查 | 在周报中标红，建议参数调整 |

> **无风险利率取值**：香港 1 月期 HIBOR（约 3.5%-4.5%，每周一盘前更新）。

#### 7.1.2 绩效指标计算规则

```
净值计算：
  账户净值 = 现金余额 + Σ(各持仓市值)
  持仓市值 = 持仓数量 × 最新价格

当日盈亏计算：
  已实现盈亏 = Σ(今日卖出成交额 - 对应买入成本 - 双边交易费用)
  浮动盈亏 = Σ((当前价 - 买入均价) × 持仓数量)
  当日总盈亏 = 已实现盈亏 + (今日浮动盈亏 - 昨日浮动盈亏)
  当日盈亏率 = 当日总盈亏 / 昨日收盘净值

最大回撤计算（滚动20日）：
  ① 记录过去20个交易日每日收盘净值序列
  ② 峰值净值 = MAX(净值序列)
  ③ 当前回撤 = (峰值净值 - 当前净值) / 峰值净值
  ④ 最大回撤 = MAX(所有时点的回撤值)

夏普比率计算（滚动60日）：
  ① 计算过去60个交易日的日收益率序列 r[]
  ② 日均收益 = MEAN(r[])
  ③ 日收益波动率 = STDEV(r[])
  ④ 年化收益 = 日均收益 × 244（港股年交易日）
  ⑤ 年化波动率 = 日收益波动率 × √244
  ⑥ 夏普比率 = (年化收益 - 无风险利率) / 年化波动率
```

#### 7.1.3 绩效告警联动

| 告警条件 | 联动动作 |
|----------|----------|
| 最大回撤 > 10% | ① 暂停所有新开仓 ② 生成《策略复查报告》 ③ 发送紧急通知 |
| 夏普比率 < 1.0 | ① 在周报中标红 ② 建议进行参数敏感性回测 |
| 连续5个交易日亏损 | ① 降一档市场立场 ② 在盘前报告中醒目提醒 |
| 单月收益率 < -5% | ① 提前触发月度优化报告 ② 建议全面复查 |

### 7.2 策略级评估（每周五收盘后）

#### 7.2.1 策略统计指标

每周五收盘后，按**四大策略**分别统计以下指标：

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| **胜率** | 盈利交易次数 / 总交易次数 × 100% | 含已平仓交易 |
| **盈亏比** | 平均盈利金额 / 平均亏损金额 | 绝对值计算 |
| **贡献度** | 该策略盈亏 / 账户总盈亏 × 100% | 可为负数 |
| **平均持仓天数** | Σ(各交易持仓天数) / 交易次数 | 分盈利/亏损统计 |
| **最大单笔盈利** | MAX(各交易盈利金额) | 金额和百分比 |
| **最大单笔亏损** | MIN(各交易盈亏金额) | 金额和百分比 |
| **期望收益** | 胜率 × 平均盈利 - (1-胜率) × 平均亏损 | 每笔交易的数学期望 |

#### 7.2.2 策略健康度评级

根据统计指标，对每个策略给予健康度评级：

| 评级 | 条件 | 标记 |
|------|------|------|
| **优秀** | 胜率 ≥ 50% 且 盈亏比 ≥ 2.0 | 🟢 |
| **良好** | 胜率 ≥ 40% 且 盈亏比 ≥ 1.5 | 🟡 |
| **一般** | 胜率 ≥ 30% 且 盈亏比 ≥ 1.0 | 🟠 |
| **衰退** | 胜率 < 30% 或 盈亏比 < 1.0 | 🔴 |

#### 7.2.3 策略衰退自动处理

```
衰退检测与自动处理规则：

阶段一：预警（连续10笔交易胜率 < 30%）
  → 自动降低该策略仓位权重至原来的 50%
  → 在周报中标红预警
  → 记录降权时间和原因到 optimization_log.json

阶段二：暂停（连续20笔仍未改善，即20笔胜率仍 < 30%）
  → 暂停该策略的信号生成
  → 在所有报告中标注该策略"已暂停"
  → 等待月度优化报告中的复核建议或用户手动恢复

恢复规则：
  → 用户可通过指令手动恢复：如"恢复策略一"
  → 恢复后以 50% 权重运行 10 笔，若胜率 ≥ 35% 则恢复全量权重
  → 若恢复后 10 笔胜率仍 < 30%，再次暂停并建议参数调整
```

#### 7.2.4 周度策略报告输出格式

```

┌─────────┬──────┬──────┬──────┬──────┬──────┐
```

### 7.3 每日交易复盘（17:00 收盘总结）

#### 7.3.1 逐笔信号回溯分析

每日 17:00 收盘总结中，对当日所有已完成交易（含止损单）逐笔进行信号回溯：

| 回溯维度 | 内容 |
|----------|------|
| **买入信号快照** | 买入时各策略子条件的具体数值（如 VR=2.3, RSI=62, MACD DIFF=0.15），从 `signal_snapshots.json` 中提取 |
| **卖出原因分类** | 止盈 / 止损（静态/移动/跳空/横盘）/ 策略信号卖出 / 应急风控 / 时间止损 / 用户手动 |
| **持仓周期对比** | 实际持仓天数 vs 策略预期持仓周期（策略一8-15日/策略二5-10日/策略三3-8日/策略四2-5日） |
| **利润捕获率** | 最大浮盈 vs 实际了结盈亏，计算 `利润捕获率 = 实际盈亏 / 最大浮盈 × 100%` |
| **滑点分析** | 指令价格 vs 用户回报的实际成交价，记录滑点金额和方向 |

#### 7.3.2 每日复盘报告格式

```
🔍 今日交易复盘（{date}）

已完成交易：{n} 笔
  ┌────┬──────────┬──────┬──────┬──────┬──────────┬──────┐
  │ #  │  标的    │ 策略  │ 盈亏  │ 浮盈峰值│ 利润捕获率│ 卖出原因│
  ├────┼──────────┼──────┼──────┼──────┼──────────┼──────┤
  │ 1  │ 00700.HK │ 策略一│ +3.2%│ +5.8% │   55%    │ 移动止损│
  │ 2  │ 09988.HK │ 策略二│ -4.1%│ +1.2% │   N/A    │ 静态止损│
  └────┴──────────┴──────┴──────┴──────┴──────────┴──────┘

📝 复盘要点：
  - {stock1}：买入时RSI={val}偏高，后续回调导致止损，归类为"追高买入"
  - {stock2}：移动止损在+5.8%处触发，但随后继续上涨至+9%，利润捕获率偏低

```

#### 7.3.3 未执行信号回顾

每日复盘还需包含"错失机会回顾"：

- 回看当日观察列表中达到触发条件但因以下原因未执行的标的：
  - 仓位已满 → 标注"仓位受限"
  - 市场立场限制 → 标注"立场限制"
  - 用户跳过 → 标注"用户跳过"
  - 其他原因 → 具体说明
- 计算"假如执行了"的虚拟收益/亏损（截至收盘价）
- 若连续5个交易日存在"因仓位满而错过盈利机会"，建议用户考虑增加资金或调整仓位上限

### 7.4 错误模式自动识别与纠正

#### 7.4.1 错误模式库

AI 须维护一个**错误模式库**（存储在 `./portfolio/error_patterns.json` 中），持续从交易记录中识别以下 8 类常见错误模式：

| # | 错误模式 | 检测规则 | 自动纠正措施 |
|---|----------|----------|-------------|
| 1 | **追高买入** | 买入价 > 当日 VWAP × 1.015 且最终亏损 | 后续同策略信号强制要求买入价 ≤ VWAP + 0.5% |
| 2 | **过早止盈** | 止盈卖出后 5 日内股价继续上涨 > 10%（连续发生 **3 次**以上） | 建议移动止损回撤比例从 5% 放宽至 7%，或延长持仓周期 |
| 3 | **止损过窄** | 止损触发后 3 日内股价反弹回成本价以上（连续发生 **3 次**以上） | 建议静态止损从 -7% 放宽至 -8%，或改用 ATR 动态止损 |
| 4 | **逆势抄底** | 市场立场为"中性观望"或"持币防守"时，策略四触发但最终亏损 > **2 次** | 临时提升策略四触发门槛（RSI < 15 且 KDJ.J < -10） |
| 5 | **同标的反复亏损** | 同一只股票 30 日内亏损交易 ≥ **3 次** | 将该标的加入 60 日冷却名单，期间不生成任何买入信号。**注意**：此 60 日冷却名单不受策略四"无视冷却期"豁免权覆盖——即使策略四极端反转信号触发，冷却名单内的标的仍不可交易（详见 §1.6 冷却机制叠加规则） |
| 6 | **行业集中踩雷** | 同一恒生二级行业 20 日内亏损交易 ≥ **5 次** | 临时将该行业仓位上限降至 15%（原30%减半），持续 20 个交易日 |
| 7 | **频繁交易损耗** | 单周交易手续费占账户净值 > **0.5%** | 次周所有策略的最小盈利阈值提高 50% |
| 8 | **事件驱动误判** | 策略三触发后亏损，且事后确认公告解读有误 | 增加"公告冷静期"：策略三信号需延迟 1 个交易日执行 |

#### 7.4.2 错误模式生命周期管理

```
错误模式生命周期：

发现阶段：
  ① 每日收盘后（17:00），扫描 trade_log.json 中最近 30 日交易记录
  ② 按上述 8 类模式的检测规则逐一匹配
  ③ 新发现的模式 → 写入 error_patterns.json，状态设为 "active"

生效阶段：
  ④ 自动纠正措施立即生效（autoApply: true）
  ⑤ 在下一份报告中通知用户：
     "⚠️ 检测到错误模式：{模式名}，已自动执行纠正措施：{措施描述}"
  ⑥ 每次该模式再次触发 → 累加触发次数，更新最近触发日期

解除阶段：
  ⑦ 若连续 **30 个交易日**未再触发该模式 → 状态设为 "resolved"
  ⑧ 自动纠正措施解除，参数回归默认值
  ⑨ 在报告中通知：
     "✅ 错误模式 [{模式名}] 已连续30日未触发，纠正措施已解除"
  ⑩ 变更记录写入 optimization_log.json
```

### 7.5 月度策略优化报告（每月最后一个周五 18:30）

#### 7.5.1 参数敏感性回测

基于过去 **60 个交易日**的全部交易记录，对每个策略的核心参数进行虚拟回测：

| 策略 | 核心参数 | 回测范围 | 评估指标 |
|------|----------|----------|----------|
| **策略一** | VR阈值、HH15振幅阈值、RSI区间 | VR: 1.5~3.0 步长0.1 | 虚拟胜率、盈亏比 |
| **策略二** | 回调位置(MA30/MA60)、缩量比(0.5~0.7) | 缩量比: 0.4~0.8 步长0.05 | 虚拟胜率、盈亏比 |
| **策略三** | 业绩阈值(营收/净利增速)、跳空幅度 | 营收: 10%~30% 步长5% | 虚拟胜率、盈亏比 |
| **策略四** | RSI超卖阈值、KDJ.J阈值、跌幅阈值 | RSI: 12~25 步长1 | 虚拟胜率、盈亏比 |

```
参数优化建议生成规则：
  ① 计算每组参数下的"虚拟"信号触发点和"虚拟"盈亏
  ② 若某参数的优化值连续 2 个月优于当前值 > 15%（胜率或盈亏比提升）
  ③ 则生成参数调整建议，标注 confidence 等级（HIGH/MEDIUM/LOW）
  ④ HIGH：连续3个月优化值一致 且 提升 > 20%
  ⑤ MEDIUM：连续2个月优化值一致 且 提升 > 15%
  ⑥ LOW：仅当月数据显示优化值提升 > 15%
```

#### 7.5.2 市场环境-策略匹配度分析

```
分析步骤：
  ① 将过去60日按市场立场分段统计天数
     - 激进做多：{n1}天  谨慎做多：{n2}天
     - 中性观望：{n3}天  持币防守：{n4}天

  ② 统计每个市场立场阶段中各策略的表现
     - 胜率差异：策略A在不同立场下的胜率对比
     - 盈亏比差异：策略A在不同立场下的盈亏比对比

  ③ 识别"策略-环境不匹配"模式，例如：
     - "策略一（趋势突破）在谨慎做多阶段胜率仅18%，建议在此阶段禁用策略一"
     - "策略二（回调低吸）在激进做多阶段盈亏比达3.2，建议提升其仓位权重"

  ④ 输出建议清单（附具体数据支撑）
```

#### 7.5.3 止损有效性分析

```
分析步骤：
  ① 统计所有止损交易的"止损后走势"
     - 止损后 5 日股价变化
     - 止损后 10 日股价变化
     - 止损后 20 日股价变化

  ② 有效性判定：
     - 若 > 60% 的止损交易在止损后 10 日内反弹回成本价以上
       → 判定"止损过于敏感"，建议放宽止损幅度
     - 若 > 40% 的止损交易在止损后继续下跌 > 10%
       → 判定"止损有效且必要"，维持当前设置
     - 若两者均不满足
       → 判定"止损幅度基本合理"，无需调整

  ③ 分类统计：
     - 按止损类型（静态/移动/跳空/横盘）分别分析有效性
     - 按策略来源分别分析止损触发频率
```

### 7.6 参数热更新

#### 7.6.1 更新方式

所有策略参数存储在 `./portfolio/strategy_params.json`（运行时参数）中。用户可通过以下两种方式不停机调参：

**方式一：自然语言**
```
将静态止损调整为-5%
移动止损激活调为+8%
单只上限调为12%
恢复默认参数
```

**方式二：JSON 格式**
```json
{
  "type": "PARAM_UPDATE",
  "updates": [{
    "category": "strategy",
    "param": "策略一.VR_threshold",
    "newValue": 2.5,
    "reason": "回测建议"
  }]
}
```

#### 7.6.2 可调参数范围

| 参数类别 | 可调参数 | 默认值 | 调整范围 | 调整示例 |
|----------|----------|--------|----------|----------|
| **止损** | 静态止损比例 | -7% | -5% ~ -10% | "将静态止损调整为-5%" |
| **止损** | 移动止损激活阈值 | +10% | +5% ~ +20% | "移动止损激活调为+8%" |
| **止损** | 移动止损回撤比例 | 5% | 3% ~ 8% | "回撤比例改为4%" |
| **仓位** | 单只上限（核心池） | 15% | 8% ~ 20% | "单只上限调为12%" |
| **仓位** | 单只上限（观察池） | 8% | 3% ~ 12% | "观察池上限调为5%" |
| **仓位** | 最低现金比例 | 10% | 5% ~ 30% | "现金比例提高到20%" |
| **应急** | 恒指减仓阈值 | -2.5% | -1.5% ~ -6% | "减仓阈值调为-3%" |
| **应急** | 恒指清仓阈值 | -4% | -3% ~ -8% | "清仓阈值调为-4.5%" |
| **筛选** | 核心池市值门槛 | 80亿 | 30亿 ~ 200亿 | "市值门槛降为50亿" |
| **筛选** | 日均成交额门槛 | 1500万 | 500万 ~ 5000万 | "成交额门槛调为1000万" |
| **数据** | 盘中监控间隔 | 30分钟 | 1分钟 ~ 60分钟 | "刷新间隔改为15分钟" |
| **策略** | 过度交易日上限 | 8次 | 4次 ~ 15次 | "日交易上限调为6次" |

#### 7.6.3 参数更新执行规则

```
参数更新流程：
  ① 解析用户输入（自然语言 → 匹配参数表 / JSON → 直接解析）
  ② 校验参数值是否在允许范围内
     - 超出范围 → 提示用户调整范围并给出允许范围
     - 在范围内 → 继续
  ③ 更新 portfolio/strategy_params.json（运行时参数文件）
     - 注意：不修改 config/strategy_params.json（默认值文件）
  ④ 记录变更日志到 optimization_log.json：
     {
       "type": "PARAM_UPDATE",
       "timestamp": "{now}",
       "param": "{param_name}",
       "oldValue": "{old}",
       "newValue": "{new}",
       "source": "user_request" | "auto_correction" | "optimization_report",
       "reason": "{reason}"
     }
  ⑤ 立即生效，在下一份报告中确认新参数值
  ⑥ 输出确认消息：
     "✅ 参数已更新：{param_name} {old} → {new}，立即生效"
```

#### 7.6.4 恢复默认参数

当用户请求"恢复默认参数"时：

```
恢复流程：
  ① 重置 portfolio/strategy_params.json 的 overrides 字段为空对象 {}
     - 注意：不是用 config/strategy_params.json 直接覆盖运行时文件
     - 运行时文件始终保持覆盖层结构（{overrides: {}, last_updated: ...}）
     - overrides 清空后，系统自动回退到 config/strategy_params.json 中的默认参数
  ② 清除所有错误模式纠正措施中的参数覆盖（但保留错误模式记录）
  ③ 更新 portfolio/strategy_params.json 的 last_updated 为当前时间
  ④ 记录恢复操作到 optimization_log.json
  ⑤ 输出确认：
     "✅ 所有参数已恢复默认值（覆盖层已清空），错误模式纠正措施中的参数覆盖已清除"

注意：恢复默认参数不会影响以下内容：
  - 错误模式库的识别记录（仅清除参数覆盖，不清除模式记录）
  - 策略衰退状态（降权/暂停状态需手动恢复）
  - 股票冷却名单（需等冷却期自然结束）
```

### 7.8 边界情况处理

| 场景 | 处理规则 |
|------|----------|
| 交易记录不足60日（新用户） | 夏普比率和月度报告使用实际可用天数计算，标注"数据量不足，结果仅供参考" |
| 某策略当月无交易记录 | 该策略评估标注"N/A - 本期无交易"，不参与健康度评级 |
| 所有策略同时衰退 | 紧急通知用户，建议暂停所有交易，等待月度优化报告 |
| 用户拒绝所有优化建议 | 尊重用户选择，记录拒绝原因，下月报告中仍包含持续跟踪 |
| 参数热更新与自动纠正冲突 | 用户手动设置优先级高于自动纠正，但需提醒用户"当前参数已覆盖自动纠正建议" |
| 错误模式库膨胀（> 20条活跃模式） | 按触发次数排序，优先展示 Top 5 高频模式，其余折叠 |
| 月度报告生成时数据源不可用 | 基于本地缓存数据生成，标注"部分数据可能存在滞后" |
| 策略恢复后再次衰退 | 暂停时间翻倍（从等待月度报告改为等待两个月度报告周期） |
